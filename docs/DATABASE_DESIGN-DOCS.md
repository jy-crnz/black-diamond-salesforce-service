# Black Diamond Salesforce Service — Storage & Schema Design Documentation

This document outlines the analytical data lakehouse architecture, data schema boundaries, and processing-engine configurations for the storage and streaming layers of the Salesforce Extraction microservice.

## 🛠️ Data Lakehouse Stack & Infrastructure

| Component | Technology | Purpose |
|-----------|-----------|---------|
| **Cold Storage (Raw Zone Data Lake)** | MinIO Object Store | S3-Compliant Cloud Storage |
| **Real-Time Event Broker** | Apache Kafka | Distributed Append-Only Log Ledger |
| **Serialization File Layout** | Apache Parquet | High-Performance Columnar Format |
| **Data Compression Engine** | Snappy | Optimized for ultra-low CPU overhead and high streaming speed |
| **In-Memory Manipulation Frame** | Pandas & PyArrow | In-Memory Processing Core |

## 📐 Global Ingestion & Payloading Conventions

To maintain pipeline uniformity and audit readiness, every record generated by the extraction loop is structured into a unified, two-tier transactional schema (meta and record blocks) before being marshaled to downstream engines.

### 1. Structural Metadata Block (`meta`)

Tracks the operational telemetry of the extraction phase to enforce data lineage:

| Field | Type | Description |
|-------|------|-------------|
| `source` | String | Hardcoded string tracking origin domain (e.g., `"salesforce"`). |
| `object` | String | CRM module identifier string target (e.g., `"Contact"`, `"Account"`). |
| `org_id` | String | Client cluster organization tier tracking code identifier. |
| `scan_id` | String | Unique transactional UUID string mapping back to Core orchestrator commands. |
| `sf_job_id` | String | Active Salesforce Bulk API 2.0 query transaction block pointer. |
| `page` | Integer | Numerical batch segment index identifier tracking the dataset chunk. |
| `extracted_at` | ISO8601 Timestamp | UTC timestamp tracking pipeline ingestion runtime boundary. |

### 2. Payload Record Block (`record`)

Contains the absolute CRM data block attributes. Every entity dynamically maintains these core tracking elements alongside standard custom data dimensions:

| Field | Type | Description |
|-------|------|-------------|
| `Id` | String | The unique 18-character, case-sensitive Salesforce identifier (used as the primary immutable identifier key). |
| `CreatedDate` | Timestamp | System audit timestamp tracking record declaration inside the CRM. |
| `LastModifiedDate` | Timestamp | Target delta audit timestamp tracking mutations (used as the baseline sorting attribute for pipeline deduplication loops). |

## 📊 Storage Layout & Partition Matrices

### 1. MinIO Data Lake Directory Hierarchy (Raw Cold Storage)

Raw record blocks are compiled directly into columnar Parquet objects and saved using Hive-style deterministic path partitioning to ensure highly efficient analytical folder scanning:

```
s3://glynac-raw-zone/
└── {org_id}/
    └── {scan_id}/
        └── {object_lower}/
            └── page_001.parquet  <-- Snappy-compressed binary file block
```

#### Parquet Target Data Layout Specs

| Attribute Field Name | Parquet Storage Data Type | Invalidation Constraint | Description / Context Rules |
|----------------------|--------------------------|------------------------|------------------------------|

| `Name` | UTF8 / String | NOT NULL | Complete Account/Contact Entity Full Record Text Name. |
| `Email` | UTF8 / String | NULLABLE | Core email address value block. |
| `Phone` | UTF8 / String | NULLABLE | Telephone metadata payload string data. |
| `CreatedDate` | UTF8 / String | NOT NULL | Normalized ISO8601 creation timestamp string. |
| `LastModifiedDate` | UTF8 / String | NOT NULL | Normalized ISO8601 mutation timestamp string. |

## ⚡ Engineering Workarounds & Critical Decisions

### 1. Zero-Disk In-Memory Parquet Buffering

**Problem:** Writing multi-megabyte staging files down onto local ephemeral container disk drives introduces heavy disk I/O bottlenecks and raises security container compliance issues.

**Resolution:** The NormalizationService handles binary conversions strictly inside volatile RAM memory layouts. Utilizing an un-bounded `io.BytesIO()` binary loop, PyArrow compiles dataframes straight into bytes before shipping them to MinIO, completely avoiding local storage footprints.

### 2. Type-Safe List Chunk Deduplication

**Problem:** The Salesforce Bulk API yields a stream of rows via Python's `csv.DictReader` list definitions, while internal storage optimization frameworks demand strict Pandas DataFrames. Running static class-based updates across mixed parameter states breaks data pipelines.

**Resolution:** The pipeline decouples filtering into an isolated DeduplicationService. The orchestrator passes raw `List[Dict[str, Any]]` objects, safely builds a local DataFrame to prioritize fresh records using `sort_values(by="LastModifiedDate", ascending=False)`, runs a `drop_duplicates(subset=["Id"])` clean sweep, and casts the deduplicated records back to structured lists (`to_dict(orient="records")`) before streaming down-sink.

### 3. PyArrow Metadata Normalization & Type Uniformity

**Problem:** The PyArrow memory engine fails with a critical schema exception when processing nested rows containing implicit timezone objects mixed with null string parameters.

**Resolution:** To ensure complete schema determinism during Parquet serialization, the normalization pipeline loops across the active dataframe columns before file export, converting system timestamps and date domains into unified string formats:

```python
for column in df.columns:
    if column.endswith("Date") or column == "SystemModstamp":
        df[column] = df[column].astype(str)
```
